# 雪倫　AI 情報警報器　設計規格書

> 一隻盯死特定來源、平常沉默、只在「真的有新發表」時才出聲的 AI 情報警報器。透過 Telegram 推送。

---

## 一、核心精神

它是**警報器**，不是**報紙**。

- 平常沒新東西，就閉嘴。不為了「看起來有在運作」每天硬擠訊息。
- 它一響＝有大事，這個「沉默本身就是訊息」的特性是整隻 bot 的價值核心。
- 原則一路貫穿：要正確不要速度、要監看不要大海撈針、平常不一定講話。

---

## 二、架構總覽

四個零件串起來：

> 觸發（排程／手動丟主題）→ 抓資料（盯來源 RSS／GitHub）→ 整理（Claude 寫成詳細白話的報告）→ 推送（Telegram）

另外有一個獨立的**來源健康監控**（見第七節），每週掃一次所有來源還活不活，壞了主動回報、但不自動改網址。

放在現有的 `AI-BOT` repo，獨立資料夾：`AI-BOT/sharon/`。與米蘭達（`mandy/`）平行擺放。共用層（`shared/`）等做到第二隻、看得出重複時再抽，現在不蓋。

---

## 三、零件一：觸發

新開一隻獨立 bot（去 @BotFather 拿新 token），跟米蘭達分開。

兩種觸發並存：

- **自動跑**：用排程器定時掃所有來源。跑在自己 VPS 用 cron；想省事用雲端排程（GitHub Actions 排程／cloud function scheduler）。
- **手動丟主題**：在 Telegram 打字觸發。收訊息接法（Webhook 或 Polling）沿用米蘭達那套，已有經驗。

---

## 四、零件二：來源清單（最關鍵的一塊）

分三層。**優先用官方原生 RSS，第三方代抓的 feed 是二手管道、要標記心裡有數。所有網址接 bot 前都要實際打開確認還活著。**

> 下表為 2026 年 5 月實際查證後的修訂版。標記說明：
> ✅ 官方原生、可直接用　⚠️ 二手或需接前再確認

## 五、零件三：整理成報告

### 用哪隻 AI（已定案）

> **Claude Sonnet 4.6**，程式中模型字串填 `claude-sonnet-4-6`。

理由：雪倫的工作是「把抓到的新聞寫成詳細白話的報告」，這是寫作任務、不是高難度推理。Haiku 太陽春、寫出來偏乾；Opus 太奢侈、貴一倍多但對寫新聞報告幫助有限；Sonnet 文筆夠好、價格適中，是官方建議的一般用途預設選擇。雪倫平常沉默、有東西才發，用量本來就不大，成本幾乎無感。

> 補充：整隻雪倫只用到這一隻 AI，而且只在最後「寫字」這一步用。前面「找資料、數數、判斷符不符合五分類」全是寫死的程式在做（例如用 feedparser 抓 RSS），不需要任何 AI。雪倫是「盯死固定來源」的警報器，不是「滿世界搜尋」的搜查員，因此不需要額外的 AI 去找資料。

### 分工原則（最重要）

> **讓 AI 寫字，不要讓 AI 做主。**
> 凡是「決定／判斷」的事用寫死的規則，AI 只負責把內容寫詳細、白話。

- **數量＝程式數數（寫死規則，不會錯）**：有幾個符合條件的重點，就發幾則。
- **內容＝Claude 來寫（它擅長的）**：每則寫得詳細、白話。

### 輸出格式：拆開一則一則發

```
🔔 前言（一則訊息：今天偵測到幾則、大概是什麼）
↓
重點一（一則訊息：詳細、白話講清楚這件事）
↓
重點二（一則訊息）
↓
…以此類推
```

每則內容建議含：標題 → 重點（詳細白話）→ 為什麼值得看 → 原文連結。

### 篩選規則：五個分類當「鏡頭」

只有符合這五類的新消息才發。

1. 新模型與功能
2. API 相關資訊
3. 圖片／影像功能
4. 價格與可用性
5. agent 與工具串接能力

**兩條鐵則：**

- 這五類是「篩子」不是「配額」。沒東西的類就空著、不發。**絕不每類硬擠一則**，否則又變回每天硬講話的報紙。
- 一個公告只發一則，就算它同時符合多類（如「Gemini 上線 API 且降價」跨 1／2／4 類）→ 一則講完、標註跨哪幾類，不重複發。

### 爆量處理（如 Google I／O 一天 50 公告）

- 不發流水帳。前言講「一次發了一堆，以下是篩出的 N 個你會在意的」，只發符合五類的重點。
- 可設單次上限（如最多 5 則），其餘併進前言。
- 篩選靠寫死的「五分類規則」，不靠 AI 主觀判斷你的喜好——你在意的（如 API 變動）就一定不會被漏掉。

## 六、開工順序

1. 用現有 `AI-BOT` repo，開 `sharon/` 資料夾。（本規格書即放於此）
2. @BotFather 開新 bot 拿 token。
3. 逐一驗證上面所有 RSS／GitHub 網址還活著（特別是標 ⚠️ 的）。
4. 先做「自動掃 RSS → 抓到新文章 → 推原始標題＋連結」最小可動版本，跑通整條線。
5. 再加上 Claude（Sonnet 4.6）整理成詳細白話報告。
6. 再加篩選規則（五分類）與拆則發送。
7. 再處理爆量上限。
8. 最後加上來源健康監控（見第七節）。

> 第一版別追求完美。先讓它跑通、會響，再慢慢加聰明。

---

## 七、來源健康監控（每週自動體檢）

> 目的：解決警報器最危險的失敗模式——「壞掉」跟「沒事」從外面看一模一樣。一個靠沉默傳遞訊息的 bot，如果某來源默默死掉，雪倫會以為「這家最近沒發」而繼續沉默，於是漏掉的消息永遠不會被發現。健康監控就是為了讓「壞掉」一定會被看見。

### 分工（已定案）

- **偵測壞掉 → 雪倫自動做**（程式做得準）。
- **修好壞掉 → 雪倫把問題丟給你，由你拍板**（換新網址需要判斷，只有人能做）。

> 為什麼不讓雪倫「自己修」：「修」的本質是換一個正確的新網址，而新網址只存在於外部世界（官方公告、GitHub README、別人的文章），程式算不出來、只能用猜的，很容易猜錯甚至猜到假官方站。更危險的是，若放任雪倫自動改來源，你就失去「它現在到底在盯哪裡」的掌控——這跟「要正確不要速度、認準真官方來源」的核心精神直接打架。所以雪倫負責永遠不會忘記提醒你，你負責換對地方。

### 機制

- 為每個來源記錄「上次成功抓到東西的時間」。
- 每週掃一次（用排程器跑，與主流程的排程分開或合併皆可）。
- 某來源若連續 N 天完全沒有任何新項目（N 值待定，初版可抓 14 天），雪倫主動推一則提醒，例如：
  > ⚠️ Qwen 來源已 14 天沒有任何更新。可能是它真的沒發，也可能是 feed 死了，請確認來源是否需要更換。
- 提醒裡要講清楚「可能真沒發、也可能壞了」兩種情況，不要直接斷定壞掉，避免誤報造成你白忙。
- 雪倫**不會**自動更換任何來源網址，一律等你確認後手動改設定。

### 注意

- N 天的門檻要按來源特性微調：大廠（OpenAI）發得勤，14 天沒消息很可疑；冷門來源或週更的 The Batch，門檻可以放寬，否則一直誤報。
- 二手代抓來源（Olshansk 那條）因為三家綁一起，一旦它整個斷線，會三家同時觸發提醒——這是預期行為，正好提醒你那條總管道掛了。
